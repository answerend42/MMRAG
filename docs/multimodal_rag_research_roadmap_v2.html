2｜多模态 RAG 解决了什么问题？

知识不只存在于文本

图表、表格、截图、流程图、公式、页面布局、照片、视频帧都可能是回答问题所需的 context。

减少 OCR / parsing 损失

传统文本化流程可能丢掉布局、视觉关系、图表趋势、区域定位；视觉检索保留页面原貌。

把企业私有知识作为外部记忆

RAG 的本质是把符合问题要求的 context 在推理时引入，知识更新更多发生在知识库/索引层。

支持可追溯答案

企业任务需要的不只是答案，还需要文档、页码、表格、图表区域、证据链和责任源。

支持跨模态任务

例如设备照片 + 维修手册、财报图表 + 文字披露、UI 截图 + 操作文档、扫描件 + 表格。

从 model-centric 回到 data-centric

高可用瓶颈常在数据类型、质量、权限、更新、元数据、检索覆盖和用户反馈，而不只是模型大小。

定义建议：多模态 RAG = 查询、知识库、检索结果、证据呈现或生成输出中，至少一个关键环节跨越文本以外模态。重点不是“是否使用 MLLM”，而是“是否引入了符合问题要求的跨模态 context”。

3｜多模态 RAG 的评估方法补充

层级建议指标为什么重要
数据 / 知识库覆盖率、源权威性、版本、权限、元数据完整性、OCR 损失、视觉元素可索引性企业 RAG 的失败常来自“没有正确数据”或“数据不可检索”。
检索Recall@k、MRR、nDCG、证据页命中、跨模态 query-doc 匹配、table/chart/image-heavy 子集表现先看正确证据有没有进入候选集，再看排位。
Rerank / 证据选择hard negative 分离、top-k 压缩后证据保留率、latency/cost、top-3/top-5 evidence accuracy决定最终送入模型的 context 是否干净、集中、可用。
生成answer correctness、faithfulness、context utilization、hallucination、abstention、冲突证据处理答案是否正确，以及是否真的依赖检索证据。
Citation / grounding文档级、页级、块级、图表区域级 citation precision/recall多模态企业 RAG 的 trust 依赖“能不能验证”。
HCI / 业务任务任务完成率、用时、验证负担、信任校准、用户控制、失败恢复、真实工作流采纳率高可用不是 benchmark 分数，而是用户能否更好完成工作。

4｜HCI / 企业视角：你这个方向很值得补

企业沉淀了什么数据？

文档、制度、合同、报告、手册、SOP、财报、图表、流程图、截图、扫描件、照片、视频帧、数据库、日志、知识图谱。每种数据都对应不同解析、分块、索引、权限、更新、引用方式。

研究问题：数据类型和数据质量如何决定 RAG 架构选择？

企业人员如何使用 RAG？

查询、核验、总结、对比、合规审查、排障、培训、销售支持、客服响应。不同用户对 citation、置信度、速度、可控性、解释方式的需求不同。

研究问题：怎样的证据呈现能降低验证负担并校准信任？

数据类型决定架构

企业知识库中文本、表格、图表、截图、扫描件的比例，如何决定 text-RAG、vision-RAG、fusion-RAG 的选择？

证据呈现影响信任

页码、区域高亮、source rank、置信度、冲突证据展示，会怎样改变专家验证行为？

用户反馈反哺数据治理

点击、追问、纠错、弃用行为能否帮助发现知识缺口、过期文档、错误 chunk、权限问题？

5｜多模态 RAG 一定需要多模态 LLM 吗？

不一定。下面这个四象限能澄清“多模态 RAG”和“MLLM”的关系。

A. 文本化 MM-RAG

图片/PDF/表格 → OCR/caption/解析 → text retriever → text LLM。适合 OCR 质量高、视觉布局不关键的场景。

B. 多模态检索 + 文本生成

用 CLIP/ColPali 等视觉检索定位图页，再把证据转成文字给 text LLM。适合视觉用于定位、推理可文本化。

C. 文本检索 + MLLM 阅读

先用文本/元数据检索文档或页码，再把页面图像、图表、截图交给 MLLM 阅读。适合答案依赖视觉细节。

D. 全链路 MM-RAG

多模态 query → 多模态 retriever/reranker → MLLM/Agent → 视觉 citation。适合复杂文档、图表、布局、多页推理。

判断规则：如果所需 evidence 可以无损转成文本，text LLM 足够；如果关键证据依赖视觉关系、布局、图表、区域定位或图像细节，就需要 MLLM 或视觉证据验证器。

6｜补充：Hybrid retrieval 和 Reranker

Hybrid retrieval

第一阶段召回：BM25/keyword + dense vector + sparse vector + metadata/permission/time filter + RRF/score fusion。目标是让正确证据进入候选集。

Reranker

第二阶段精排：对 query-context pair 做更细判断，处理 hard negatives，把正确证据推到 top-3/top-5，减少塞给 LLM 的噪声。

能不能去掉？

可以在高置信、简单查询、正确证据稳定 top-3/top-5 时跳过；复杂多跳、候选相似、需要视觉 citation 或高风险业务时建议 adaptive rerank。
